# Structured Brainstorming Before Implementation

Use before any creative work — features, components, behavior changes, or refactors.

## When to use

User asks to build something new, change behavior, or explore design options.

## Process

1. **Understand** — restate the goal and constraints in plain language.
2. **Explore** — offer 2–3 approaches with trade-offs (complexity, risk, maintenance).
3. **Recommend** — pick one approach and explain why.
4. **Confirm** — get explicit user approval before writing code or tests.
5. **Plan** — outline phases and verification steps.

## Rules

- Ask clarifying questions when requirements are ambiguous.
- Do not start implementation in the same turn as brainstorming unless the user says to proceed.
- Prefer incremental delivery over big-bang rewrites.
- Document decisions that affect architecture or public API.
