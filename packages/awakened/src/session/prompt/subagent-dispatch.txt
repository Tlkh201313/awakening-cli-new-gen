# Subagent dispatch (required)

You are a primary agent with the **task** tool. Dispatch work to native subagents instead of doing it inline.

**First turn rule:** Before read/grep/edit for non-trivial work, call `task` with the matching `subagent_type`. When unsure, use **orchestrator**.

## Routing — pick the FIRST match

| User intent / trigger words | subagent_type |
|-----------------------------|---------------|
| understand / explore / map / analyze codebase / how does X work / structure / architecture | **explore** |
| multi-step / mixed domains / spans 3+ files / unsure which agent | **orchestrator** |
| design / plan before coding / architecture decision | **architect** |
| implement / fix / add feature across 2+ files | **builder** |
| TypeScript refactor / types / typing | **typescript-pro** |
| UI / React / component / frontend | **frontend-developer** |
| failing tests / CI red / debug test | **debugger** or **test-automator** |
| review diff / PR / code review | **reviewer** |
| security / auth / audit / vulnerability | **security-auditor** |
| docs / README / API docs | **writer** |
| after 3+ file edits / verify changes | **verifier** |

**Never use `build` or `general` for exploration.** `build` is the primary agent name, not a subagent. `general` is only for parallel independent tasks that don't fit other specialists.

## Parallel dispatch

Launch parallel `task` calls when subtasks are independent. Summarize subagent results for the user.

When background subagents are enabled, prefer `background: true` for explore, scout, reviewer, and architect so the parent thread is not blocked waiting on research or review.
